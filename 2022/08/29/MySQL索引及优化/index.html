MySQL索引及优化

发表于2022-08-29|更新于2022-09-03
|阅读量:

本部分内容从官方文档搬运，仅用于学习、记录。以官方文档为准:MySQL :: MySQL 8.0 Reference Manual :: 8.3.1 How MySQL Uses Indexes

概念

索引用于快速查找据有特定的列值所在的行。如果没有索引，MySQL一定会从第一行开始，遍历全表搜索相关的行，如果表很大，那么这个步骤耗时就越久。如果表中存在索引，那么MySQL就可以快速定位到数据文件中间的查询的位置，而不用去查询所有的数据。

这点很好理解，想象你拿到了一本字典，现在让你不查看目录找到某一个字所在的位置，那就只能从头开始一页页查找。如果这个字还是多音字，为了不避免遗漏，需要从头找到尾。如果给字典加上索引(目录)，那么我们直接在目录中找到这个字所在的页码，然后到对应的页码找到这个字，效率比从头翻到尾快上不少。

分类

主键索引(唯一且非空)

唯一索引（唯一可为空）

普通索引（普通字段索引）

全文索引（一般是char、varchar、text类型）

组合索引

数据结构

MySQL的索引用到的数据结构多种多样，可以总结为下面几点：

1、大部分的索引(主键索引、唯一索引、普通索引和全文索引)保存在B+树

扩展：为什么采用B+树？

B+树能显著减少IO次数，提高效率

B+树的查询效率更加稳定，因为数据放在叶子节点

B+树能提高范围查询的效率，因为叶子节点指向下一个叶子节点

B+ Tree Visualization (usfca.edu)

2、空间数据类型的索引采用R树

3、MEMROY搜索引擎支持利用哈希进行索引

4、InnoDB的全文索引采用倒排链表

使用索引的操作

MySQL中使用索引的操作有：

1、快速查到和WHERE语句匹配的行

2、如果有多个索引，那么MySQL通常会选择查找到的行数最少的索引

3、如果表中有多列索引，那么索引可以从任意从最左边的前缀进行查询

4、在联表时从其他表中索引。如果这两列是同样类型和大小，那么MySQL可以更有效地使用索引。

5、查找特定索引列的最大值和最小值。

6、在可用索引的最左前缀上进行排序或分组。

7、在某些情况下，可以在不查阅数据行的情况下检索值。 (覆盖索引)

索引优化

索引的字段选择规则

通常为select、update、delete的where条件列、order by、group by、distinct字段、多表join字段。

对where条件中范围查询的第一个范围条件字段建索引

多个范围条件里边的最左边的第一个范围查询用到索引，但是后边的范围查询则不会使用

主键字段不要重复建索引

大多数数据库都对主键自动建主键索引，无需针对主键再建索引

对外键关联字段建索引

对于外键关联字段，一般都应添加索引，因为它们经常用于查询

避免对大字段建索引

如果非必要，不对检索text或bolb的字段，不要索引大段文字。这样会让索引占据大量存储空间。如果一定要检索，必须采用全文检索。

不要对小型表进行检索

不要为小型数据表设置索引,，尤其是这个表经常有插入和删除操作。对这些插入和删除操作的索引维护可能比扫描表空间消耗更多的时间

定期维护或重建索引

对于经常进行数据插入、删除、更新的字段应定期维护或重建索引。

适当使用组合索引

如某个表中的字段通常是一起被查询，那么可以针对这些字段建组合索引来提高效率（一般不超过3个）。建立联合索引时，必须将区分度更高的字段放在左边。数据库查询时通过最前面的条件逐个过滤，第一个条件过滤越多，后面的条件需要过滤的就越少。

有大量重复值的字段不宜建索引

区分度不高的列单独创建索引的优化效果很小，并且频繁的更新则会让索引的维护成本高。

索引个数不要太多

不必要的索引会浪费空间和时间，因为MySQL需要耗费时间确定要使用哪些索引。索引还会增加插入、更新和删除的成本。

尽量对比较短的值进行索引

创建使用该列的前n个字符的索引，可以使索引文件更小。如果查找的值超过索引前缀长度，则使用索引排除不匹配的行，并在剩余的行继续查找。
